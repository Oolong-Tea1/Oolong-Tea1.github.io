S3 存储桶权限配置实战：如何防止数据误公开
626 字 2 分钟


Amazon S3 是企业最常用的云存储服务之一，用来存放日志、备份、业务文件甚至前端静态资源。
然而，S3 存储桶误公开 是最常见的安全事故之一，许多企业因为错误配置导致敏感数据被所有人访问，造成严重后果。

今天我们来手把手演示 如何正确配置 S3 存储桶权限，避免“裸奔”。

一、S3 默认安全机制

  1. 默认私有
    • 新建的 S3 存储桶默认只有创建者能访问。
  2. 阻止公共访问（Block Public Access）
    • AWS 提供了全局和存储桶级别的“阻止公共访问”选项。
    • 默认勾选，防止误操作。

二、创建存储桶时的权限配置

  1. 打开 S3 控制台 → Create bucket
  2. 输入存储桶名称，选择区域
  3. 保持 Block Public Access（阻止公共访问） 全部勾选
  4. 直接下一步

👉 说明：
除非你明确需要托管静态网站，一般不要取消这个选项。

三、存储桶权限管理方式

在 存储桶 → Permissions 标签 下，可以看到三种主要的权限配置方式：

  1. 存储桶策略（Bucket Policy）
    • JSON 格式，灵活但容易误操作。
    • 例子：可以只允许某个 IAM 角色访问。
  2. 访问控制列表（ACL）
    • 早期功能，AWS 已不推荐使用。
  3. IAM 策略
    • 在用户/角色层面控制访问，安全性更高。

四、常见误区：Public Bucket

很多安全事故来自于用户在 Bucket Policy 中写了：

这意味着 任何人都能访问存储桶中的文件！
AWS 控制台会在你保存时警告：This bucket has public access。

五、最佳实践

  1. 始终保持 Block Public Access 全部勾选。
  2. 如果需要对外公开文件，建议：
    • 使用 预签名 URL（Presigned URL）
    • 或者托管在 CloudFront CDN，再设置访问控制。
  3. 定期使用以下工具检查安全：
    • Amazon Macie → 自动发现敏感数据
    • Trusted Advisor → 扫描公共访问风险

六、总结

  • S3 默认是私有的，但配置不当可能导致数据裸奔。
  • 千万不要轻易取消 Block Public Access。
  • 最佳做法是通过 IAM 控制访问，而不是开放存储桶本身。

✅ 建议：今天就去检查一下你现有的存储桶，确认“阻止公共访问”是否开启。
