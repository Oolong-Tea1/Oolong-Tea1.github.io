AWS 配额与申请扩展指南
833 字 2 分钟

在日常使用 AWS 的过程中，你可能会遇到这样的问题：

  • 想要启动更多的 EC2 实例，却提示“配额不足”；
  • 想创建多个弹性 IP（EIP），结果发现最多只能有 5 个；
  • 部署多个环境时，VPC 数量受限。

这些情况其实都是 AWS 配额（Service Quotas） 的限制。本文将带你全面理解配额，并手把手教你如何申请扩展。

一、什么是 AWS 配额（Service Quotas）

AWS 配额指的是 AWS 账号在某个服务下能使用的最大资源数量或速率。

  • 软限制（Soft Limit）：默认值较低，可以申请提升（例如 EC2 实例数量、EIP 数量）。
  • 硬限制（Hard Limit）：无法提升，受 AWS 架构约束（例如单个安全组最多规则数）。

常见默认配额示例：

  • EC2：单个区域内默认 20 个实例
  • EIP：每个区域最多 5 个
  • VPC：每个区域默认 5 个
  • RDS：默认最多 40 个数据库实例

👉 不同服务、不同区域的配额各不相同。

二、为什么要关注配额

  1. 避免上线受阻：如果在业务高峰时才发现资源不够用，会严重影响进度。
  2. 合理规划架构：提前了解配额，能避免因限制导致的部署失败。
  3. 企业合规需求：大规模应用通常会涉及数十个 VPC、上百个实例，需要提前扩容。

三、如何查看当前配额

AWS 提供了 Service Quotas 控制台，你可以在里面查看和管理所有服务的配额。

操作步骤：

  1. 登录 AWS 控制台，搜索并进入 Service Quotas。
  2. 选择需要查看的服务，例如 EC2。
  3. 在列表中即可看到该服务的配额详情（名称、默认值、可否调整）。

四、如何申请提升配额

如果你发现某个配额不足，可以通过控制台发起申请。

建议是在全英文界面下进行申请。

操作步骤：

  1. 打开 Service Quotas → 进入目标服务（例如 EC2）。
  2. 找到要扩展的配额（如 Running On-Demand Standard instances）。
  3. 点击 Request quota increase。
  4. 输入目标值（如从 20 提升到 50）。
  5. 提交申请后，AWS 会自动生成支持工单，等待审批。

五、常见的需要提升的配额场景

  • EC2 实例数：企业级应用需要更多计算节点
  • EIP 数量：多环境、多服务的公网访问需求
  • VPC 数量：多租户架构或区域隔离部署
  • CloudFormation 堆栈资源限制：大规模自动化部署时常遇到

六、最佳实践

  1. 提前申请
    在业务上线前至少一周提交扩容申请，避免紧急情况。
  2. 注意区域性
    配额是按 Region 区分的，比如东京区域的配额扩展不会影响新加坡区域。
  3. 监控与自动化
    使用 Service Quotas API 或 CloudWatch 定期检查配额，避免超限。
  4. 企业支持计划
    建议启用 Business/Enterprise Support，审批速度更快，更适合大规模团队。

七、总结

  • 配额是 AWS 的资源使用上限，既是保护机制，也是规划依据。
  • 查看 → 申请 → 扩展 是标准流程，通常 1–2 天内即可完成。
  • 建议所有团队在正式部署前 预估配额需求并提前申请，避免生产环境卡壳。
AWS 教程：如何升级正在使用的 EC2 实例配置

在日常使用 AWS EC2 时，可能会遇到以下情况：

  • 实例运行越来越慢，需要升级 CPU / 内存
  • 磁盘空间不足，需要扩容 EBS
  • 想要更高的网络性能

下面就来一步步讲解，如何安全地升级正在使用的 EC2 实例。

1️⃣ 升级实例类型（CPU / 内存）

步骤 1：停止实例

  • 在修改实例类型前，需要先 停止实例（注意：停止不会丢失数据，但会短暂停机）。

步骤 2：修改实例类型

  • 在弹出的对话框中选择新的实例类型，例如从 t3.micro 升级为 t3.medium。
  • 点击 Apply 保存。

步骤 3：启动实例

  • 启动实例，新的配置就会生效。
  • 登录实例，确认服务运行正常。

2️⃣ 升级存储（EBS 磁盘扩容）

步骤 1：修改卷大小

  • 进入 Elastic Block Store (EBS) → 选择挂载的磁盘 → 点击 Modify volume。
  • 增加磁盘大小，例如从 20 GB 升级为 50 GB。
  • 提交后，EBS 会自动扩容（在线操作，无需停机）。

步骤 2：调整文件系统

  • 登录 EC2 实例，执行以下命令，让操作系统识别新的空间：

Linux 示例：

Windows 示例：

  • 打开 磁盘管理 → 右键分区 → 扩展卷。

3️⃣ 升级前的注意事项

  • 快照备份：在任何升级前，先对 EBS 卷创建快照。
  • 实例类型兼容性：有些老实例需要停机迁移到新硬件才能升级。
  • 最小权限原则：操作前确认 IAM 用户有 ec2:ModifyInstanceAttribute、ec2:ModifyVolume 权限。
  • 业务停机规划：实例类型升级必须停止实例，建议选择业务低峰期操作。

4️⃣ 总结

  • 升级实例类型：停机 → 修改 → 重启
  • 扩容磁盘：在线修改 EBS → 登录实例扩展文件系统
  • 记得做好快照备份，确保数据安全

通过以上步骤，你就能平滑升级正在使用的 EC2 实例配置，提升性能的同时，最大程度保证业务连续性。

AWS 中 IAM MFA 设置教程：从零开始管理用户与权限~

在 AWS 中，IAM（身份与访问管理） 是最基础也最重要的安全配置。
通过 IAM，你可以创建独立的用户、分配合适的权限，避免所有人都用 root 账号操作 AWS，从而降低安全风险。

下面，我们通过几个步骤来完成 IAM 基础配置。

1️⃣ 创建 IAM 用户

进入 IAM 控制台，选择 Add user。
输入用户名（例如 user），选择 AWS Management Console access，并设置初始密码。
这样，新建的用户就可以通过独立账号登录 AWS，而不用使用 root 账号。

2️⃣ 创建用户组（Group）并分配权限

最佳实践是通过 用户组来管理权限。
例如，创建一个 user-group 组，然后为它附加策略 AmazonEC2FullAccess。
这样，所有加入这个组的用户都能管理 EC2 实例，而不用逐个分配权限。

3️⃣ 将用户加入用户组

接下来，把 user 加入 user-group 组。
用户组的策略会立即生效，这样用户就继承了组的所有权限。

5️⃣ 开启 MFA（手机扫码登录）

为了提高账号安全性，建议给 IAM 用户启用 MFA（多因素认证）。
具体操作步骤：

  1. 打开用户的 Security credentials 页面，点击 Assign MFA device
  2. 选择 Authenticator app（推荐），系统会生成一个 二维码
  3. 打开手机上的 Google Authenticator、Microsoft Authenticator、1Password 等 TOTP 应用，扫码绑定
  4. 应用会生成一个动态的 6 位验证码（30 秒刷新一次）
  5. 在 AWS 控制台输入两次连续的验证码，验证成功后绑定完成

👉 之后，每次用该 IAM 用户登录 AWS 控制台时，除了输入密码，还需要输入手机上的验证码，相当于 手机扫码 + 动态口令登录，大大提高了账号安全性。

6️⃣ 最佳实践小结

  • 不要用 root 账号日常操作
  • 每个人都有自己的 IAM 用户
  • 使用用户组来批量管理权限
  • 给敏感操作配置 最小权限原则
  • 开启 MFA，保证账号安全
